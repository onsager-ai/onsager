## 1. Purpose

Onsager is an AI factory that produces **artifacts**. This document defines the precise semantics, data model, and invariants of the artifact concept inside Onsager. All four subsystems (Forge, Stiglab, Synodic, Ising) share the vocabulary defined here.

This document does **not** define any subsystem's implementation details, API surface, or UI shape. It answers one question only: **"What is an artifact in Onsager?"**

---

## 2. Definition

> **An artifact is a persistent, identity-stable, value-bearing thing that Onsager exists to produce.**

All four qualifiers are necessary. Each excludes a category that is frequently confused with artifact:

- **persistent** — The artifact's lifecycle is independent of any single session. Sessions end; the artifact continues. It remains referenceable until explicitly archived. This distinguishes artifact from *session output* (ephemeral, session-scoped intermediate products).
- **identity-stable** — An artifact has an ID that remains constant across its entire lifecycle. A single "HP Nova quarterly report" may be shaped by 5 sessions, each producing a new version, but it is always **one** artifact. This distinguishes artifact from *one-off output* (a new thing each session).
- **value-bearing** — Every artifact must have value for some declared consumer. An artifact with no consumer is an anomaly that must be surfaced. This distinguishes artifact from *internal scratch* (temporary data used by agents during execution).
- **thing Onsager exists to produce** — Artifacts are Onsager's **purpose**, not its means. Rules, insights, escalations — these exist to make artifacts better, not the other way around. This distinguishes artifact from *governance byproduct* (metadata generated by the governance process itself).

---

## 3. What artifact is NOT

To prevent vocabulary drift, the following are explicitly **not** artifacts:

| Concept | Why not an artifact | What it is instead |
|---|---|---|
| Session-internal `tool_use` results | Ephemeral, session-scoped, no independent ID | session-internal event |
| Synodic rules | A governance tool, not a produced value object | governance asset |
| Ising insights | An improvement signal, not a produced value object | observation |
| Factory event stream entries | Audit records, not produced value objects | factory event |
| Session transcripts | Traces of shaping, not the thing being shaped | session record |
| Agent memory / context | Execution detail, session-scoped | runtime state |

**Discrimination rule**: If a thing exists in order to make artifacts better, it is not an artifact. If a thing exists because it is what Onsager was built to deliver as value, it is an artifact.

## 4. Core data model

Each artifact is an entity with the following fields:

### 4.1 Identity

- `artifact_id` — Globally unique, stable ID. Never changes, never reused. Formatted as `art_8a3f`.
- `kind` — The artifact's type tag (see §5).
- `name` — Human-readable name. May be edited without affecting identity.
- `created_at` — Timestamp of first registration.
- `created_by` — Source of first registration (user, session, system).

### 4.2 Ownership and consumption

- `owner` — The responsible party (user or team). Every artifact **must** have an owner.
- `consumers` — Declared list of consumers (user / team / system / external endpoint). **At least one** required. An empty consumer list is an anomaly.

### 4.3 Versioning

- `current_version` — Currently active version number, monotonically increasing integer.
- `versions[]` — Version history. Each version contains:
  - `version` — integer
  - `created_at` — timestamp
  - `created_by_session` — the session ID that shaped this version
  - `content_ref` — pointer to external storage (see §6)
  - `change_summary` — semantic summary of what changed from the previous version
  - `parent_version` — usually `current_version - 1`, but branching is permitted in the data model (not the v0.1 semantics; see §10)

### 4.4 Lineage

- `vertical_lineage` — **Shaping history**: which sessions participated in which versions. Fundamentally a relation list of `session_id × version`.
- `horizontal_lineage` — **Dependency graph**: which **other** artifacts were used as inputs when shaping this artifact. Fundamentally a relation list of `(other_artifact_id, used_at_version, role)`.

Together, these two lineage types constitute the full "factory production traceability". Any artifact can answer both "who made me" and "what did I use to be made".

### 4.5 State

Each artifact lives in one explicit lifecycle state:

```
draft → in_progress → under_review → released → archived
                ↑                        |
                └────────── revise ──────┘
```

State transitions are driven by **Forge**. Each transition emits an `artifact.state_changed` event to the factory event spine. Every transition may be gated by **Synodic** (blocked or required to escalate).

Non-linear constraints:
- From `released`, an artifact cannot return directly to `draft`. It must pass through `revise` into a new `in_progress` cycle.
- `archived` is a terminal state. It is irreversible.
- Any state can transition directly to `archived` (early termination).

### 4.6 Quality signals

- `quality_signals[]` — Aggregated view of multi-source quality signals. Each entry contains:
  - `source` — signal origin (automated_test / lint / human_review / ising_inference / synodic_check / external)
  - `dimension` — dimension name (correctness / completeness / safety / readability / ...)
  - `value` — scalar score or discrete label
  - `recorded_at` — timestamp
  - `recorded_by` — the entity that produced this signal

Quality signals are **append-only**. They are never overwritten. Aggregated views ("the current overall quality of this artifact") are computed on demand by consumers.

## 5. Kind system

`kind` is the artifact's type tag. It is **typed but not closed**:

- The system ships with a set of built-in kinds: `code`, `document`, `report`, `dataset`, `config`, `api_call`.
- Users and organizations may declare custom kinds and register them in the kind registry.
- Each kind declares which **quality dimensions** it expects — for example, `code` expects `correctness` (test pass rate) and `safety` (lint), while `report` expects `completeness` and `readability`.

`kind` is not an inheritance hierarchy. It is a flat label. An artifact has exactly one kind.

**Why not closed**: Onsager should not hardcode users' business concepts. HP Nova may need a `test_execution_report` kind; a consulting firm may need an `industry_research_brief` kind. None of these should require modifying Onsager's core.

---

## 6. Storage model

**Onsager holds metadata, not content.**

- Artifact metadata (id, version history, lineage, quality signals, state) lives in Onsager's core database (PostgreSQL).
- Artifact **content** lives externally: Git, S3, Notion, Confluence, database tables, API endpoints, etc.
- The `content_ref` field is a URI plus an optional checksum pointing to the precise location and version of external content.

**Rationale**:
1. Artifact content is highly heterogeneous; unified storage is an antipattern.
2. Content already lives in users' existing systems; forced migration is a massive adoption barrier.
3. This keeps Onsager lightweight, preventing it from becoming a storage service.
4. Lifecycle management can be optimized independently (metadata persists; content follows each backend's own strategy).

**Cost**: Content may be modified externally without Onsager knowing. This is mitigated with checksums plus periodic reconciliation, but strong consistency is not pursued.

## 7. Invariants

The following invariants must be maintained by Forge. Violating any of them is a system bug:

1. **Identity conservation** — Once an `artifact_id` is assigned, it always refers to the same artifact.
2. **Version monotonicity** — `version` is monotonically increasing. Numbers are never reused.
3. **Lineage completeness** — Every version must have a traceable `created_by_session`. Anonymous versions are not permitted.
4. **Owner non-null** — Every artifact must always have an owner.
5. **Consumer non-empty** — Every artifact must always have at least one consumer (may be a system placeholder, but must be explicitly declared).
6. **State machine validity** — State transitions must follow the state machine in §4.5.
7. **Content ref immutability per version** — Once a version's `content_ref` is recorded, it cannot be modified. Changing content requires producing a new version.

## 8. Relationships to subsystems

This document does not define subsystem implementations, but the **relationship role** each subsystem has with artifacts must be explicit:

| Subsystem | Role | Actions |
|---|---|---|
| **Forge** | Owner of artifact lifecycle | Register, version, advance state, schedule shaping |
| **Stiglab** | Shaper | Runs sessions under Forge's instruction; submits shaping results |
| **Synodic** | Gate | Evaluates governance at state transitions; may block or escalate |
| **Ising** | Observer | Continuously observes production efficiency and quality; feeds back insights |

Note that **Forge is the only subsystem with write access to artifact state**. Other subsystems interact with Forge through event protocols; they do not modify artifact state directly. This single-writer constraint protects all invariants in §7.

---

## 9. Anti-patterns

The following practices are antipatterns in Onsager and should be blocked by tooling or review:

1. **Anonymous artifact** — an artifact without an owner or consumer
2. **Orphan session** — a session not bound to any artifact (sessions must declare which artifact version they are shaping)
3. **Silent content mutation** — modifying external content without producing a new version
4. **Cross-kind reuse** — changing an `artifact_id` from one kind to another
5. **Ghost lineage** — referencing an already-archived artifact as a horizontal lineage input without declaration
6. **Quality signal overwrite** — attempting to modify or delete a recorded quality signal

## 10. Open questions

Left for subsequent specs. Not answered in this version:

1. **Branching** — Is an artifact allowed to have branched versions (v3a / v3b in parallel)? The current data model allows `parent_version` to point to a non-latest version, but branching rules for merging, naming, and lifecycle are undefined. Leaning toward not supporting branches in v0.1 and adding when a real need appears.
2. **Multi-owner** — Is an artifact allowed to have multiple owners? Currently single-owner only. The responsibility dilution issues introduced by multi-owner need separate evaluation.
3. **Cross-tenant lineage** — Can horizontal lineage reference an artifact belonging to another organization? Involves permission boundaries; deferred to a multi-tenancy spec.
4. **Quality dimension governance** — Are the quality dimensions declared by a kind mandatory? If a `code` artifact has no `correctness` signal, can it enter the `released` state?
5. **Reconciliation cadence** — Frequency and strategy for consistency checks between external content and Onsager metadata.
6. **Identity migration** — How identity is established when a user wants to "import" an existing external thing as an artifact.

## 11. Glossary

| Term | Definition |
|---|---|
| **Artifact** | See §2 |
| **Kind** | The artifact's type tag; extensible |
| **Version** | A concrete snapshot in an artifact's lifecycle |
| **Vertical lineage** | Shaping history: which sessions made which versions |
| **Horizontal lineage** | Dependency graph: which other artifacts this artifact references |
| **Shaping** | The modifying action Stiglab sessions perform on an artifact |
| **Gating** | Synodic's governance judgment at state transition points |
| **Content ref** | URI + checksum pointing to external storage |
| **Quality signal** | A single append-only record about artifact quality |
