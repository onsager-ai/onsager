01 · 总览

Plan Compiler Pipeline

0.2 的三层 pipeline。Spec Plan（谁先做谁后做）+ Workflow Library（每种 kind 的标准流程）→ Plan Compiler 编译成 Execution Plan（不可变实例）→ Substrate 调度器执行产 artifact。整个 pipeline 跟数据库查询编译同构：Spec Plan = Logical Plan，Execution Plan = Physical Plan，Plan Compiler = Query Compiler，Substrate = Query Executor。

怎么读

  • 顶部两个输入并列：Spec Plan（项目唯一）+ Workflow Library（库可复用）
  • review_wf 灰色淡显——库里有但这次没用，强调库的容量>当前 plan 需求
  • Plan Compiler 三步明写：lookup → instantiate → connect，无 LLM 介入
  • 输出 Execution Plan：外层骨架=Spec Plan 依赖，内层节点=Workflow Library 模板复制
  • s2 和 s3 都用 impl_wf → 容器内结构完全一致（岛内同构，见下图）
02 · 简洁性原则

N 个同构岛

每种 spec kind 对应一个 workflow。同 kind 的 spec 共享同一个 workflow shape（岛内同构）；跨 kind 的 workflow 形状不同（岛间异构）。简洁性来自把变化都装在 spec 的 kind 字段——Plan Compiler 只需 lookup。统一 entry/exit 契约让跨岛连接像 function call 一样直接。

怎么读

  • 把 s_impl_a 和 s_impl_b 像素级对照：节点数、名字、位置、颜色、内部箭头——完全一致。这是岛内同构
  • s_design 跟两个 impl 故意不同（研究/草图/评审/终稿 vs 准备/编码/测试/发布）。这是岛间异构
  • 黑色粗箭头 = Spec Plan 依赖，从 design 的 exit（final）分叉给两个 impl 的 entry（prep）
  • 跨岛连接靠统一 entry/exit 契约——任意 workflow 入出口都用 ArtifactId，跟 function signature 同构
03 · 类型系统

Provenance 流

substrate 的"类型系统"。每条边携带 artifact 的 provenance 信息：Deterministic（脚本/确定性来源）或 Uncertain（agent/LLM 来源）。Verify executor 是唯一允许把 Uncertain 升级成 Deterministic 的节点。下游边若声明 requires_deterministic=true，未经 Verify 的 Uncertain 上游无法连过去——substrate 拒绝加载。

怎么读

  • 边的颜色 = 流过这条边的 provenance：绿色实线 Det / 红色虚线 Unc
  • agent 节点产 Uncertain（LLM 输出本质不确定）→ 出边变红
  • run_tests 是 Verify（紫框）—— Uncertain 进，Deterministic 出，颜色变化点
  • ship 入边 requires_deterministic=true，加粗——如果 Verify 没通过，整条 Workflow 拒绝执行
  • 这是 substrate 5 条 kernel 不变量中最核心的两条（不变量 1 + 不变量 2）
04 · 组合机制

SubWorkflow 递归

workflow 内部可以放 SubWorkflow 节点引用另一个 workflow——Beer recursion 在 substrate 层直接成立。外层 code_review 节点的 executor 是 SubWorkflow，引用 review_wf；运行时它展开为内层完整 workflow。同样结构可以无限嵌套：workflow → workflow → workflow ...

怎么读

  • 外层 feature_wf：4 个节点的普通 workflow（plan → develop → code_review → ship）
  • code_review 紫色框 = SubWorkflow executor，workflow_ref: review_wf 表明引用
  • 内层 review_wf 是它自己的完整 workflow（gather → assess → approve），其中 approve 又是 Verify
  • 内外两层是同一种 workflow 抽象——递归不是特殊机制，是统一接口的副产品
  • 对应 Beer 的 "contains, and is contained in"——每一层都是 viable system
05 · 第二种公民

Observer 旁路

Observer 是 substrate 的第二种公民（除 Workflow 外）。它不在 Workflow 里，订阅 spine 事件流，异步产 QualitySignal / Insight / Alert。Observer 不阻塞主流程——它发现问题只能产 insight 让人决定，没法直接停 Workflow。映射 VSM S3*（audit channel）。Ising 现有 analyzer 全部归为 Observer instance。

怎么读

  • 实线（横向）= Workflow 主流程，靠 artifact 串
  • 虚线（纵向）= 事件流——所有节点 fire-and-forget 把执行事件 emit 到 spine
  • spine 是 append-only event store，不阻塞 Workflow emit
  • 3 个 Observer 各订阅不同 pattern（subscribes 字段），产不同输出（emits 字段）
  • VSM 对应：Workflow = S1/S2/S3（操作），Observer = S3*（审计，audits, doesn't manage）
06 · 综合

全貌剖面图

把上面 5 张图全部装进一个具体运行实例。Spec Plan 有 2 个 spec（一 design 一 impl），Plan Compiler 编译产 Execution Plan，里面 s_impl 包含 SubWorkflow（review）展开成内层 workflow，内层有 Verify（approve）把 Uncertain 升级成 Deterministic。所有节点 emit 事件到 spine，3 个 Observer 旁路读。5 个紫色 callout 串起来。

5 个 callout 对应 0.2 5 个核心机制

  • ① Spec Plan 外层骨架——谁先做谁后做（人决定）
  • ② Plan Compiler 编译——deterministic 3 步算法
  • ③ SubWorkflow 嵌套——同一接口无限递归
  • ④ Verify 升级点——唯一允许 Unc → Det
  • ⑤ Observer 旁路——substrate 第二种公民，VSM S3*
